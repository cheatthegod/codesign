- **序列预测（seq）：** 大幅提升。v1 由于数据 bug 导致前 23K 步 seq=0，之后以 0.5 倍损失权重和 1/r 权重爆炸进行训练。v2 从第 0 步开始以完整权重正常训练 seq，稳态 seq 损失降低 39%。

## 6. 发现并修复的 Bug

### 6.1 [严重] BUG-1：序列损失在 23,000 步内为零

**根本原因：** 训练数据缓存（`local_data/pdb_cache/*.pt`）在 2026-04-14 01:14-01:27 期间被重建。旧缓存文件缺少 `gt_seq` 字段。`runner/train.py:193-196` 中的回退逻辑将缺失的 `gt_seq` 替换为全 UNK token：

```python
gt_seq = input_feature_dict.get("gt_seq")
if gt_seq is None or gt_seq.shape[0] != n_prot:
    gt_seq = torch.full((n_prot,), PRO_STD_RESIDUES.get("UNK", 20))
```

模型迅速学会对每个位置预测 UNK，从 step 2,550 到 step 26,000 序列损失为 0.0000。当新缓存（包含真实 `gt_seq`）在 step 26,050 出现时，seq 损失飙升至 9-287，训练随即崩溃。

**影响：** 23,000 步（约占第一训练阶段的 88%）的序列训练完全浪费。序列解码器实质上从 step 26K 才开始真正学习。

**v2 修复：** 新缓存已包含 `gt_seq`。经检查验证：所有缓存 .pt 文件中 `gt_seq` 字段均存在，包含正确的氨基酸索引（0-19）。

### 6.2 [严重] BUG-2：噪声配置死代码

**根本原因：** `disco/data/train_data_pipeline.py` 中的 `noise_sequence()` 函数硬编码 `mask_rate = torch.rand(1)`（均匀 0-1），未接收或使用配置中的 `seq_noise_min` / `seq_noise_max` 值。这些配置项存在但是死代码。

**v2 修复：** 修改 `noise_sequence()` 添加 `noise_min` 和 `noise_max` 参数：

```python
def noise_sequence(seq_tokens, mask_token_idx=MASK_TOKEN_IDX, noise_min=0.0, noise_max=0.95):
    r = torch.rand(1, device=seq_tokens.device)
    mask_rate = (noise_min + (noise_max - noise_min) * r).squeeze()
```

修改 `runner/train.py` 传递配置值：

```python
noised_seq, mask_rate, masked_positions = noise_sequence(
    gt_seq,
    mask_token_idx=MASK_TOKEN_IDX,
    noise_min=configs.noise.get("seq_noise_min", 0.0),
    noise_max=configs.noise.get("seq_noise_max", 0.95),
)
```

### 6.3 [严重] BUG-3：1/r 权重爆炸

**根本原因：** SUBS/MDLM 序列扩散损失使用权重 `1/(1 - mask_rate)`。当 `mask_rate ~ U(0, 1)` 时，该权重具有**无穷方差**：

| mask_rate | 权重 = 1/(1-mask_rate) | P(权重 > W) |
|---|---|---|
| 0.99 | 100 | 每个样本 1% |
| 0.999 | 1,000 | 每个样本 0.1% |
| 0.9999 | 10,000（截断最大值） | 每个样本 0.01% |

每个优化器步骤有 32 个样本，至少一个样本权重 > 100 的概率为 **26.8%** —— 大约每 4 个优化器步骤就有一个被极端异常值主导。

**v2 修复：** 设置 `seq_noise_max = 0.95`，将最大权重限制为 `1/(1-0.95) = 20`。效果：
- 平均权重：10.0 → 3.2
- 最大权重：10,000 → 20
- 权重标准差：133.9 → 3.2
- 每个优化器步骤出现 spike 的概率：26.8% → 0%

### 6.4 [中等] CONFIG-1 至 CONFIG-6

六个配置参数偏离 DISCO 论文（表 S1），全部修正为论文值或接近值。详见第 2.3 节完整对比表。

### 6.5 [轻微] 基础设施：日志缓冲

**根本原因：** `nohup torchrun ... > file 2>&1` 导致 torchrun 进程中继管道的输出缓冲。训练进度在数分钟到数小时内不可见。

**修复：** 在 `runner/train.py` 中直接添加行缓冲的 `FileHandler`：

```python
fh_stream = open(log_file, "w", buffering=1)  # 行缓冲
file_handler = logging.StreamHandler(fh_stream)
```

## 9. 建议

### 9.1 后续步骤

1. **运行推理：** 使用 `disco_ema_step_100000_final.pt` 在一组酶靶标测试集上评估生成质量（scTM、scRMSD、序列恢复率等指标）。

2. **对比 EMA 检查点：** 在留出验证集上比较 80K、90K 和 100K 的 EMA 检查点——最佳 EMA 检查点不一定是最后一个。

3. **清理非 EMA 检查点：** 如需释放磁盘空间，完整检查点（每个 5.9 GB）仅在恢复训练时需要：
   ```bash
   rm train_output_conditional/checkpoints/disco_step_*.pt  # 节省约 65 GB
   ```

### 9.2 进一步训练的可能性

- 在 100K 步时损失仍在缓慢下降。额外训练 50K 步（扩展至 150K）可能带来约 5-10% 的 seq 改善，但回报递减。
- 如需扩展：使用 `load_checkpoint_path: ./train_output_conditional/checkpoints/disco_step_100000.pt`（含优化器状态的完整检查点），设置 `max_steps: 150000`。

### 9.3 未来训练的架构注意事项

- `load_strict: false` 设置会静默跳过不匹配的权重。未来训练建议使用 `load_strict: true` 并显式处理架构变更，避免类似 seq=0 的静默失败。
- `sequence_diffusion_loss` 中的 `1/r` 权重可通过重要性采样（如 MDLM 论文推荐）进一步改进，而非简单截断，尽管当前方法已经有效。
